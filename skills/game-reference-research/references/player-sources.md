# Player Sources

Use this reference for publicly available reviews, community screenshots, discussion boards, wiki pages, streams, and lets-play videos. Route user-supplied first-party sessions and private player feedback to `$synthesize-playtest-feedback`.

For a fixed set of supplied public comments, analyze that set before gathering more. Record platform, date, version, query, or selection method only when it changes how the result should be interpreted.

If origin is unresolved, ask only when public versus first-party context would change routing or attribution. Never infer that separate comments represent distinct people.

## Source Types

Reviews:

- Good for pain points, delight, confusion, pacing complaints, content value, replayability.
- Weak for exact mechanics unless repeated and specific.

Community screenshots:

- Good for real gameplay states, late-game density, player-created layouts, modded/workshop examples.
- Must be labeled as UGC and lower confidence than official media.

Discussions:

- Good for recurring issues, FAQ friction, balance complaints, control problems, common beginner mistakes.
- Beware outdated posts after patches.

Wiki/guides:

- Good for mechanics, recipes, numbers, content lists, progression details.
- Check version/date and cite as community-maintained unless official.

Videos/streams:

- Good for interaction timing, onboarding, feel, pacing, and moment-to-moment decisions.
- Prefer official trailers for intended presentation and recent gameplay videos for actual play.

## Review Mining

Look for repeated phrases around:

- "confusing", "grindy", "addictive", "too slow", "hard to read", "clunky", "satisfying".
- Tutorial/onboarding problems.
- UI management burden.
- Difficulty spikes.
- Endgame boredom or content shortage.
- Controller/Steam Deck issues.

Summarize useful patterns and counterexamples, not an exhaustive sentiment report. Public comments can suggest a pain point or payoff; they do not establish population frequency without a valid denominator.

## Safety And Rights

- Do not use community screenshots or store images as project assets without rights.
- Use public media as visual/design references and keep source attribution.
- Avoid quoting long review text. Paraphrase and link when possible.
